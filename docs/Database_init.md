# Database_init.md — Supabase SQL Setup (Stage 1 Advanced)

Run this SQL once in your Supabase project's **SQL Editor**
(Dashboard → SQL Editor → New query). It creates the `todos` table,
turns on Row Level Security (RLS), and adds the four policies that
make sure each user can only ever see and change their own rows.

> This stage does **not** include any groups feature. There is no
> `groups` table and no `group_id` column — that will be added in
> Stage 2.

```sql
-- =====================================================
-- 1. Create the todos table
-- =====================================================
create table if not exists todos (
  id           bigint generated by default as identity primary key,
  title        text not null,
  is_complete  boolean not null default false,
  user_id      uuid not null references auth.users (id) on delete cascade,
  created_at   timestamp with time zone not null default now()
);

-- Optional but recommended: speeds up "select where user_id = ..."
create index if not exists todos_user_id_idx on todos (user_id);

-- =====================================================
-- 2. Enable Row Level Security
-- =====================================================
-- Until RLS is enabled, the anon key could read/write ANY row
-- in this table. Enabling it blocks all access by default —
-- the policies below are what re-opens access, but only for
-- each user's own rows.
alter table todos enable row level security;

-- =====================================================
-- 3 & 4. Allow users to SELECT only their own todos
-- =====================================================
create policy "Users can view their own todos"
  on todos
  for select
  using (auth.uid() = user_id);

-- =====================================================
-- 5. Allow users to INSERT only their own todos
-- =====================================================
-- "with check" runs on the row being inserted: it must have
-- user_id equal to the id of the currently authenticated user.
create policy "Users can insert their own todos"
  on todos
  for insert
  with check (auth.uid() = user_id);

-- =====================================================
-- 6. Allow users to UPDATE only their own todos
-- =====================================================
-- "using" controls which existing rows can be targeted;
-- "with check" controls what the row is allowed to look like
-- after the update. Both must match auth.uid() = user_id so a
-- user can't update someone else's row or reassign a row to
-- another user_id.
create policy "Users can update their own todos"
  on todos
  for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

-- =====================================================
-- 7. Allow users to DELETE only their own todos
-- =====================================================
create policy "Users can delete their own todos"
  on todos
  for delete
  using (auth.uid() = user_id);
```

## How to run this

1. Go to your Supabase project dashboard.
2. Open **SQL Editor** in the left sidebar.
3. Click **New query**, paste the SQL block above, and click **Run**.
4. Open **Table Editor** and confirm you now see a `todos` table with
   the columns `id`, `title`, `is_complete`, `user_id`, `created_at`.
5. Open **Authentication → Policies** and confirm the `todos` table
   shows 4 policies (select, insert, update, delete).

## Quick sanity check

After running the app and signing up as two different test users
(e.g. `alice@example.com` and `bob@example.com`), each user should
only ever see the todos they personally created. If either user could
see the other's todos, double-check that:

- RLS is actually **enabled** on the table (step 2 above), and
- All four policies exist and reference `auth.uid() = user_id`.
